## What this project is

C3S is a Spring Boot website builder that serves CMS-driven sites without requiring the CMS to run on the host. A request comes in, C3S looks up the site by virtual host, loads its structure/templates from a "master" backend, fetches content from a "content" backend, and renders templates server-side. There is no separate frontend — the application *is* the public site for every site it hosts.

Full project documentation lives at https://github.com/limburgie/c3s/wiki.

## Build / run

- Build: `./mvnw clean package` (or `mvn clean package`). The `openapi-generator-maven-plugin` runs during `generate-sources` and writes Paginea client code to `target/generated-sources/openapi` from `src/main/resources/paginea.json` — `mvn clean` removes it, so a fresh build regenerates.
- Run locally: `mvn spring-boot:run` or `java -jar target/c3s.jar`. The default profile uses `c3s.registry.type=LOCAL` from `application.properties`; the developer-specific profile is activated via `-Dspring.profiles.active=peter` and pulls config from `application-peter.properties` (gitignored, contains DB credentials and reCAPTCHA secret).
- No test suite exists in `src/test`. The only tests under `target/` are auto-generated by the OpenAPI plugin and should not be edited.

## Architecture

### Strategy-pattern plugin model

Three pluggable abstractions share the same shape: an interface, multiple `@Service` implementations, an enum identifying each implementation, and a `*Factory` that injects `List<Interface>` and picks the right one by enum value. When adding a new backend, follow this pattern — Spring auto-discovery does the wiring.

| Abstraction | Selector enum | Implementations |
|---|---|---|
| `RepositoryRegistry` (registry/service) | `RepositoryRegistryType` | `LocalRepositoryRegistry` (properties), `DatabaseRepositoryRegistry` (JPA/MySQL), `PrismicRepositoryRegistry` |
| `MasterService` (master/service) | `RepositoryType` | `WebserverMasterService` (currently the only impl — reads `c3s.json` over HTTP/file URL) |
| `ContentService` (content/service) | `RepositoryType` | `ContentfulContentService`, `PrismicContentService`, `MockContentService`, plus a Paginea slot under `content/service/paginea` populated by OpenAPI codegen |
| `TemplateParser` (master/templateparser) | `TemplateEngine` | `FreemarkerTemplateParser` (only impl, despite the enum listing `VELOCITY`/`THYMELEAF`) |

### Request flow

`PageController` (controller/PageController.java) is the single `@RestController` and handles every URL via catch-all mappings (`@Order(2)`):

1. `findMasterRepository(request.getServerName())` resolves the virtual host through the active `RepositoryRegistry`.
2. `MasterServiceFactory.forRepositoryConnection(...)` (request-scoped) instantiates and `initialize()`s the matching `MasterService` — this loads `c3s.json` describing pages, templates, forms, locales.
3. If the master config declares a content repository, `ContentServiceFactory` does the same for the content backend.
4. `PageRenderer` builds the FreeMarker context with `api`, `site`, `request`, `uri`, `inserts`, `i18n`, `cart` and parses the page (or its template chain — templates can extend other templates, with named `inserts` substituted in).
5. Output is post-processed via Jsoup to inject `c3s-edit-url.js`.

Special routes handled before the catch-all: `/assets/**`, `/favicon.{ico,svg}`, `/submit` (form POST), `/c3s/{contentItemId}` (redirect to CMS edit URL), `/sitemap.xml`.

### Request-scoped state via ThreadLocals

Per-request state is propagated through `ThreadLocal`s rather than method parameters: `LocationThreadLocal` (locale + timezone), `RequestUriThreadLocal` (current URI without locale prefix), `ShoppingCartThreadLocal`. Code reading these assumes `PageController` has set them earlier in the request — preserve that ordering when adding new entry points.

### Content API abstraction

`ContentApi` + `QueryBuilder` (content/service/domain/) are a backend-agnostic content query language exposed as `api` in templates. Each backend implementation translates calls into native CMS queries. When changing this interface, every `*ContentApi`/`*QueryBuilder` must be updated in lockstep — there is no shared base class.

### Paginea content backend

Paginea is the in-progress content backend. Its package `be.webfactor.c3s.content.service.paginea` is empty in source — the API and model classes are generated from `src/main/resources/paginea.json` into `target/generated-sources/openapi/...` by `openapi-generator-maven-plugin` (configured in `pom.xml`). The hand-written `Paginea*ContentService` glue (analogous to `ContentfulContentService`) does not yet exist; adding the backend means writing those wrappers around the generated API client.
